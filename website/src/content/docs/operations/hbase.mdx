Actionbase uses HBase as its primary storage backend. This guide covers HBase setup and configuration for Actionbase.

## Overview

HBase is a distributed, scalable, big data store that provides:

- **Reliability**: Proven in production at scale
- **Scalability**: Horizontal scaling capabilities
- **Consistency**: Strong consistency guarantees
- **Performance**: Low-latency random read/write access

## Installation

### Using Docker

For local development:

```bash
docker run -d \
  --name hbase \
  -p 2181:2181 \
  -p 16000:16000 \
  -p 16010:16010 \
  -p 16020:16020 \
  -p 16030:16030 \
  harisekhon/hbase:latest
```

### Production Setup

For production deployments, follow HBase cluster setup best practices:

1. Set up ZooKeeper ensemble
2. Configure HBase cluster nodes
3. Set up HDFS (if using HDFS as storage)
4. Configure region servers
5. Set up monitoring and alerting

## Configuration

### Actionbase Configuration

Configure Actionbase to connect to HBase:

```yaml
datastore:
  type: hbase
  hbase:
    zookeeper:
      quorum: zookeeper1:2181,zookeeper2:2181,zookeeper3:2181
    client:
      retries: 3
      timeout: 30000
      operationTimeout: 30000
```

### HBase Configuration

Key HBase settings for Actionbase:

```xml
<!-- hbase-site.xml -->
<configuration>
  <property>
    <name>hbase.zookeeper.quorum</name>
    <value>zookeeper1,zookeeper2,zookeeper3</value>
  </property>
  <property>
    <name>hbase.client.retries.number</name>
    <value>3</value>
  </property>
  <property>
    <name>hbase.client.operation.timeout</name>
    <value>30000</value>
  </property>
</configuration>
```

## Table Design

Actionbase automatically creates and manages HBase tables with optimized design:

### Row Key Design

- Optimized for range scans
- Supports efficient prefix lookups
- Balanced distribution across regions

### Column Families

- Organized for optimal read/write patterns
- Compression enabled by default
- TTL configured based on data retention policies

### Region Management

- Automatic region splitting based on size
- Region distribution across cluster
- Automatic rebalancing

## Performance Tuning

### Write Optimization

- Enable compression (Snappy or LZ4)
- Configure memstore flush size
- Tune write buffer size

### Read Optimization

- Enable block cache
- Configure cache size
- Tune scan caching

### Region Server Tuning

- Configure heap size appropriately
- Tune GC settings
- Monitor region server metrics

## Monitoring

### Key Metrics

Monitor the following HBase metrics:

- **Region server health**: Uptime, request count
- **Read/write latency**: p50, p99, p99.9
- **Region distribution**: Balance across nodes
- **Storage utilization**: Disk usage, compaction status

### Tools

- HBase Web UI: `http://<region-server>:16010`
- HBase Shell: `hbase shell`
- Monitoring tools: Prometheus, Grafana

## Backup and Recovery

### Backup Strategies

- Snapshot-based backups
- Export/Import for data migration
- Replication for disaster recovery

### Recovery Procedures

- Restore from snapshots
- Point-in-time recovery
- Region server failure recovery

## Troubleshooting

### Common Issues

**Connection timeouts**: Check ZooKeeper connectivity and network latency

**Region server failures**: Monitor heap usage and GC logs

**Slow reads**: Check block cache hit rate and region distribution

**Slow writes**: Monitor memstore flush frequency and compaction status
