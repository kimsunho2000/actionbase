Actionbase uses a Property Graph model to represent data. This document explains how Actionbase encodes and stores graph data in HBase.

## Property Graph Model

Actionbase uses a Property Graph model where data is represented as:

- **Vertices (Nodes)**: Entities with unique IDs and properties
- **Edges (Relationships)**: Connections between vertices with properties
- **Flexible Schema**: Properties can be added or modified without schema migrations

```
User (vertex)
  ├─ id: "user123"
  └─ properties: { name: "John", email: "john@example.com" }

User --[follows]--> User (edge)
  ├─ source: "user123"
  ├─ target: "user456"
  └─ properties: { createdAt: 1234567890, weight: 0.8 }
```

## Write-time Optimization

Actionbase pre-computes indexes and counters during writes to enable ultra-fast reads. When an edge is written, Actionbase:

1. Stores the edge state (EdgeState)
2. Creates index entries for scan queries (EdgeIndex)
3. Updates counters for count queries (EdgeCounter)

This allows read operations to simply retrieve pre-computed data from storage.

## Storage Encoding

Actionbase stores edge data in HBase using multiple Row Types. Each Row Type serves a specific query purpose.

### Row Types

#### EdgeState

Stores the current state of edges for Get queries.

**Key Structure:**
```
[4-byte hash] + [1-byte + source] + [1-byte + table code] + [1-byte + -3] + [1-byte + target]
```

**Value Structure:**
- active: Edge active status (Boolean)
- version: Edge version (Long)
- properties: Property map with version per property
- createdAt: Creation time (Long)
- deletedAt: Deletion time (Long)

#### EdgeIndex

Stores index entries for Scan queries.

**Key Structure:**
```
[4-byte hash] + [1-byte + directed source] + [1-byte + table code] + [1-byte + -4] + 
[1-byte + direction] + [1-byte + index code] + [(1-byte + N) * # index values] + [1-byte + directed target]
```

**Value Structure:**
- version: Edge version (`Long`)
- properties: Property map (`Map<String, Any>`)

#### EdgeCounter

Stores counters for Count queries.

**Key Structure:**
```
[4-byte hash] + [1-byte + directed source] + [1-byte + table code] + [1-byte + -2] + [1-byte + direction]
```

**Value Structure:**
- count: Edge count (Long)

### Row Key Encoding

All Row Keys follow this common pattern:

```
[4-byte hash] + [1-byte + source] + [1-byte + table code] + [1-byte + Row Type Code] + [additional fields...]
```

- **Hash**: 4-byte xxhash32 for HBase Region distribution
- **Row Type Code**: Unique negative code (-1, -2, -3, -4) to identify data type

### Value Encoding

Actionbase uses Byte Headers to maintain type information:

- Each value: `1-byte type information + actual value`
- Type information distinguishes NULL, STRING, INT, FLOAT, JSON, etc.
- ASC/DESC headers for sort order
- EdgeState stores version per property; EdgeIndex stores version per edge

## Data Flow

### Write Path

```
Client → Server → Engine → WAL → HBase → CDC
```

1. Lock acquisition (EdgeLock)
2. State read (EdgeState)
3. State transition and mutation creation
4. WAL write (Kafka)
5. HBase storage: Update EdgeState, create/delete EdgeIndex, increment/decrement EdgeCounter
6. CDC write (Kafka)

### Read Path

```
Client → Server → Engine → HBase → Response
```

- **Get**: Retrieve EdgeState by key
- **Scan**: Range scan EdgeIndex by index
- **Count**: Retrieve EdgeCounter value
